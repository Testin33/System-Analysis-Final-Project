Chapter 5: Structural Modeling & Class Design

5.1 Overview

The class diagram for the Story Scripting Editor is organized into four packages: patterns, models, services, and ui. This separation reflects a clear layered architecture where domain entities, behavioral patterns, business logic, and interface concerns are kept independent of one another.

5.2 Package: patterns

This package defines the structural foundation for the Observer design pattern used throughout the system.

Observable is an abstract class that maintains a private list of registered observers (- observers: List[Observer]) and exposes three public methods: add_observer(observer), remove_observer(observer), and notify_observers(). Any class that extends Observable automatically gains the ability to broadcast state changes to all registered listeners without knowing anything about them. In this system, Project is the only class that extends Observable.

Observer is an interface with a single method contract: + update(project: Project): void. Any class that implements this interface agrees to receive and process project state changes whenever they are broadcast. In this system, Dashboard is the sole implementor.


5.3 Package: models

This package contains all domain entities that represent the core data of a narrative project.

Project is the central entity of the system. It extends Observable, so every mutation it undergoes - adding or removing a story, character, or location - ends with a call to notify_observers(), triggering an automatic update to all registered observers. A project holds four types of assets through composition relationships: a list of Story objects, a list of Character objects, a list of Location objects, and exactly one BackgroundStory. Composition means these assets cannot exist outside of a project context. The remove operations (remove_story, remove_character, remove_location) return a boolean to indicate whether the deletion was allowed or refused.

Story represents a single narrative thread within a project. Each story holds an ordered list of Scene objects and carries a lifecycle state (StoryState). The version_label and parent_version_id attributes enable story branching: a writer can call create_version(label) to derive a new Story from an existing one, producing a separate branch without modifying the original. This is how the system achieves branching while keeping the narrative within each individual story strictly linear. reorder_scenes(order: List[str]) accepts a list of scene IDs and rearranges the story sequence, supporting both drag-and-drop and typed input.

StoryState is an enumeration with three values - DRAFT, VALIDATED, and PUBLISHED - that models the maturity lifecycle of a story from initial writing to final release.

Scene is the atomic unit of narrative content. It holds the written text (content), a title, a short_description, an emotion_tag, and an action_tag. Scenes reference characters and locations by ID strings (character_refs, location_refs) rather than by direct object reference. This allows a writer to embed a [CHAR:ID] or [LOC:ID] token in scene text before the corresponding entity has been defined; the ID is only resolved to a real name at compile time by the StoryCompiler. The is_empty() method returns true when both title and content are blank, which the Validator uses to detect incomplete scenes.

[INSERT FIGURE 5.3 HERE: Zoom into the Story and Scene class boxes, the composition arrow between them, the StoryState enum, and the self-association on Story representing the branching chain. Caption: Figure 5.3 - Story, Scene, and StoryState]

Character represents a narrative actor within a project. Characters are global to the project and not bound to any single story. The visible boolean flag allows a writer to hide a character from the active workspace without deleting them, a requirement confirmed explicitly by the professor. The attributes dictionary accommodates writer-defined fields such as personality, emotional background, trust level, and backstory, making the model flexible without requiring a fixed schema. The is_referenced_in(scenes) method enables the delete-guard logic: a character cannot be removed if they are still referenced in any scene.

Location follows the same structural pattern as Character: it is global to the project, carries an attributes dictionary, and exposes is_referenced_in(scenes) to prevent deletion of locations that are still referenced.

BackgroundStory is a single world-building document attached to each project. It is composed of one or more BackgroundSection objects, each with a title and a content field that may contain [CHAR:ID] and [LOC:ID] tokens - the same syntax used in scenes - so that background text stays consistent with the rest of the project at compile time.

5.4 Package: services

This package contains the business logic layer of the system. Services operate on model objects but do not own them; they are stateless processors that accept model instances as input and return results.

Validator accepts a Project and returns a ValidationReport. It walks every story, scene, character, and location in the project and checks for blank fields, unresolved character or location IDs, and isolated scenes that have no sequence position. The validator never blocks the writer: it reports issues as warnings and errors but does not prevent saving or continuing work.

ValidationReport is the output object of the Validator. It holds two lists - warnings and errors - and exposes has_issues() and summary() as convenience methods for the UI to display results without processing the raw lists directly.

StoryCompiler takes an ordered Story and the parent Project and replaces all [CHAR:ID] and [LOC:ID] tokens in scene content with the real names of the corresponding character and location objects. It exposes three methods: compile() for full compilation, preview() for an in-app read without writing to disk, and export_to_file() for writing the compiled output to a file in .txt, .md, .json, or .pdf format.

SearchService provides multi-criteria filtering over scenes and characters. search_scenes(criteria) accepts a dictionary of filter keys such as emotion_tag, character_ref, or title, and returns only the scenes that match all supplied criteria simultaneously. search_characters(criteria) operates the same way over the character list. This design, confirmed by the professor, supports combining multiple filters in a single query.

Storage handles all persistence. save_project() serializes a Project to JSON and writes it to a file path. load_project() reads and reconstructs a Project from a JSON file. auto_save() is triggered automatically after every project mutation to ensure crash recovery without any manual save action from the writer.

5.5 Package: ui

Dashboard is the sole UI class represented in the class diagram. It implements the Observer interface, meaning it receives a Project instance every time the project changes and independently recalculates its metrics without the Project knowing anything about the Dashboard's internals. It exposes methods for scene count, character count, word count, character-to-scene mapping, emotion tag distribution, action tag distribution, and incomplete scene listing. The emotion tag distribution powers the emotional graph the professor described during the interview. The Dashboard never asks the Project for an update; instead it waits passively for the Observer notification, which is what keeps the two classes loosely coupled.

5.6 Key Design Decisions

Three design decisions in the class diagram are worth noting explicitly.

First, scenes reference characters and locations by ID strings rather than by object reference. This allows scenes to be created before the referenced entities exist and supports the [CHAR:ID] and [LOC:ID] token system used by the compiler.

Second, branching is modeled at the Story level via parent_version_id rather than at the Scene level. This keeps scene sequencing strictly linear within each story, which was a hard constraint confirmed by the professor: no parallel branching within a story.

Third, Validator and StoryCompiler are separate service classes rather than methods on Project or Story. This keeps the domain model clean and makes each service independently testable without needing a full project object instantiated.
