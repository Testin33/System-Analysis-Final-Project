Chapter 7: Design Patterns & Architecture
==========================================


7.1 Selected Pattern
--------------------

The Story Scripting Editor implements the Observer design pattern. This
pattern defines a one-to-many dependency between a subject object and a
set of observer objects, so that when the subject changes state, all
registered observers are notified and updated automatically without the
subject knowing anything about them.


7.2 The Problem This Pattern Solves
------------------------------------

The system has two objects whose state must always be in sync: the
Project (the data) and the Dashboard (the metrics view). The Dashboard
must reflect the current scene count, character count, word count,
emotion tag distribution, and list of incomplete scenes at all times.
Without a pattern, there are two naive options: either the Project
directly calls Dashboard.refresh() every time it changes, which couples
domain logic to UI code, or the Dashboard polls the Project periodically,
which is wasteful and introduces stale state.

Observer solves both problems. The Project broadcasts a notification
whenever its state changes, and the Dashboard listens and recalculates
independently. The Project never references the Dashboard directly. The
Dashboard never asks the Project for updates. The two classes are
completely decoupled.


7.3 Implementation in This System
----------------------------------

The pattern is implemented across two packages.

In the patterns package, Observable is an abstract class that holds a
private list of Observer references and exposes three operations:
add_observer(), remove_observer(), and notify_observers(). Observer is a
single-method interface whose only contract is update(project: Project).

In the models package, Project extends Observable. This means Project
inherits the observer list and the notification mechanism. Every mutating
operation in Project (add_story, remove_story, add_character,
remove_character, add_location, remove_location) ends with a call to
notify_observers(), which iterates the list and calls update(this) on
each registered observer.

In the ui package, Dashboard implements Observer. When registered, it
receives a Project reference every time the project changes and
independently recalculates all its metrics. It does not store a stale
copy of the project between notifications; it always works from the
freshest state passed into update().

The notification chain for any project mutation looks like this:

    Writer action in UI
        -> Project.add_story() / remove_character() / etc.
        -> Project.notify_observers()
        -> Dashboard.update(project)
        -> Dashboard recalculates metrics
        -> Storage.auto_save(project)

The result is that the writer sees live metrics in the Dashboard after
every single edit, without any manual refresh.

- Project inherits from Observable (extends arrow)

7.5 Why Observer Over the Alternatives
---------------------------------------

Three other patterns were considered and rejected.

Command was considered because it encapsulates operations as objects and
naturally supports undo/redo. It was rejected because the professor
explicitly confirmed that no revision history or undo system is required
in this project. Command would add structural complexity for a feature
that is out of scope.

Strategy was considered for the compilation and validation logic, since
it allows swapping algorithms at runtime. It was rejected because the
system has exactly one compilation algorithm and one validation algorithm.
There are no interchangeable variants, so the flexibility Strategy
provides is unnecessary here.

State was considered for managing the Story lifecycle (DRAFT, VALIDATED,
PUBLISHED). It was rejected because the lifecycle has only three states
with simple, direct transitions. A full State pattern would be
over-engineering for what a three-value enum and a change_state() method
handle cleanly.

Observer was chosen because it directly matches the real architectural
problem: keeping the Dashboard synchronized with the Project without
creating a dependency between them. It is the only pattern among the
alternatives that decouples state broadcasting from state consumption,
which is exactly what this system needs.
